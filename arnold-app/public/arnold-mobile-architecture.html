Native Wrapper — Capacitor Architecture

Why Capacitor? Arnold is already a Vite + React web app. Capacitor wraps the existing web build in an Android WebView with zero code rewrite. Kotlin plugins provide the native bridge to Health Connect. React Native would require a full rewrite; Capacitor keeps the existing ~13,000 lines of code intact.

Key New Files

hc-bridge.js
JavaScript wrapper around Capacitor.Plugins.HealthConnect. Detects web vs native environment — in browser, returns mock data or falls back to FIT upload. In Capacitor, calls the Kotlin plugin via the bridge.

import { Capacitor } from '@capacitor/core';
const isNative = Capacitor.isNativePlatform();
export const hcBridge = isNative ? Capacitor.Plugins.HealthConnect : mockHC;
hc-sync.js
Orchestrates periodic Health Connect reads. Runs on app open + every 15 minutes. Reads all data types since last sync timestamp, maps HC records to Arnold's storage format, merges nutrition (dedup), and writes to storage.js.

Key functions:
syncAll() · syncExercise() · syncSleep()
syncWeight() · syncHR() · syncNutrition()
mergeNutrition() · writeBackToHC()
HealthConnectPlugin.kt
Kotlin Capacitor plugin implementing @CapacitorPlugin. Uses androidx.health.connect.client SDK to read/write HC records. Handles permission requests, date range queries, and data type serialization to JSON for the JS bridge.

Dependency: androidx.health.connect:connect-client:1.1.0-alpha
capacitor.config.ts
Capacitor configuration. Points WebView to the Vite build output. Registers the HealthConnect plugin. Sets Android minimum SDK 26 (required for Health Connect).

{ appId: 'com.arnold.health',
  webDir: 'dist',
  plugins: { HealthConnect: {} } }

Web vs. Native — Graceful Degradation

Same codebase, two modes. The hc-bridge.js module detects the runtime environment. On the web (GitHub Pages), Arnold works exactly as today — FIT uploads, CSV imports, manual entry. On Android (Capacitor), Health Connect takes over as the data source. All existing features (EdgeIQ, race calendar, encryption, AI food log) work identically in both modes. No code duplication.

Garmin → Health Connect Data Gaps

Garmin does not write all of its proprietary metrics to Health Connect. These three data types require workarounds.

HRV (Heart Rate Variability)
GAP
Status: Garmin measures HRV nightly but does NOT write it to Health Connect.

HC Type: HeartRateVariabilityRmssdRecord exists in HC, but Garmin doesn't populate it.

Workarounds:
1. Garmin Connect API — OAuth app, rate-limited, requires developer access
2. Connect IQ widget — push HRV from watch → phone → Arnold via intent
3. Periodic CSV export — current approach, still works as fallback

Recommendation: Start with CSV fallback, then build Connect IQ widget for Phase 3.
Body Battery
GAP
Status: Garmin proprietary metric. NO Health Connect equivalent exists.

HC Type: None — no standard body battery / energy concept in HC.

Workarounds:
1. Garmin Connect API — has body battery endpoint
2. Arnold can approximate using HRV + sleep + activity load (EdgeIQ already does readiness scoring)
3. CSV export — current approach

Recommendation: Enhance Arnold's readiness score to approximate Body Battery; use CSV as supplement.
Training Effect
GAP
Status: Garmin proprietary (Firstbeat). Aerobic/Anaerobic TE scores not in HC.

HC Type: None — HC has ExerciseSessionRecord but no TE fields.

Workarounds:
1. FIT file parsing — TE is embedded in FIT activity files (keep FIT upload as option in native)
2. Arnold can estimate using TRIMP/EPOC from HR data (EdgeIQ already computes training load)
3. Connect IQ data field — push TE after each activity

Recommendation: Use Arnold's existing TRIMP calculation as proxy; keep FIT upload available for detailed analysis.
Long-term strategy: All three gaps are solvable via the Garmin Connect API (OAuth2 + REST endpoints for daily summaries, activities, body battery). This would be a Phase 3 server component — a lightweight cloud function that polls Garmin's API daily and pushes data to Arnold via push notification or background sync. Until then, Arnold's own algorithms (readiness scoring, TRIMP) provide good approximations.

Implementation Phases

1
Capacitor Shell + HC Read (Est. 2–3 weeks)
Goal: Arnold runs as an Android app, reading activity/sleep/weight/HR from Health Connect automatically.
Tasks:
• Initialize Capacitor project wrapping existing Vite build
• Write HealthConnectPlugin.kt — read-only for Exercise, Sleep, Weight, HeartRate
• Write hc-bridge.js — environment detection + native/web fallback
• Write hc-sync.js — sync orchestrator with last-sync timestamps
• Update Arnold.jsx boot sequence: call hcSync.syncAll() if native
• Modify LogDay: hide FIT upload on native, show "Last synced" indicator
• Android permissions manifest: ACTIVITY_RECOGNITION, HEALTH_CONNECT
• Test with Garmin Forerunner data via Health Connect on physical device
Outcome: Zero daily uploads for activities, sleep, weight, resting HR
2
Nutrition Sync + Write-back (Est. 2 weeks)
Goal: Cronometer food data flows into Arnold automatically. Arnold food log entries flow back to Health Connect.
Tasks:
• Add readNutrition() + writeNutrition() to Kotlin plugin
• Implement mergeNutrition() — dedup by timestamp + food name
• Source tagging: each nutritionLog entry gets source field
• Auto-tag Cronometer meals as pre/during/post workout by activity time window
• Write-back: when user logs food in Arnold, push to HC as NutritionRecord
• Update Nutrition tab to show source badges (🅒 Cronometer, 🅐 Arnold)
• Test bidirectional sync: log in Cronometer → see in Arnold → log in Arnold → see in Cronometer
Outcome: "One Body in one place" — all food from both sources visible in Arnold
3
Gap Closers + Background Sync (Est. 3–4 weeks)
Goal: Close the three Garmin gaps (HRV, Body Battery, Training Effect) and enable background sync.
Tasks:
• Option A: Connect IQ Widget — Garmin watch data field that pushes HRV + Body Battery + TE to phone after each activity sync. Requires Garmin SDK (Monkey C).
• Option B: Garmin Connect API — lightweight cloud function (Cloudflare Worker / Deno Deploy) that polls Garmin's REST API daily via OAuth2, pushes data to Arnold via Web Push notification.
• Implement Android WorkManager for periodic background HC sync (every 30 min)
• Add push notification: "New activity synced — 5.2 mi run, 485 cal"
• Enhanced readiness scoring using all three metrics natively
Outcome: True zero-touch — all data flows automatically, no manual intervention
